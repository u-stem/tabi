# Playwright テスト拡充設計

## 背景

現在 62 件の E2E テストが 16 ファイルに分散しており、主要な操作はカバーされている。
一方で経費・みやげ・投票・通知・権限制御など、機能として実装済みだがテストされていない領域がある。
本設計ではその全操作をカバーする 10 ファイルを追加し、「あらゆる操作がテストされた状態」を実現する。

## スコープ

### 対象 (新規追加)

| ファイル | 対象機能 |
|---------|---------|
| `expenses.spec.ts` | 経費管理 |
| `polls.spec.ts` | 日程投票 |
| `reactions.spec.ts` | 候補へのリアクション |
| `activity.spec.ts` | アクティビティログ |
| `notifications.spec.ts` | 通知・通知設定 |
| `souvenirs.spec.ts` | みやげリスト |
| `export.spec.ts` | エクスポートページ |
| `print.spec.ts` | 印刷ページ |
| `roles.spec.ts` | 権限ベースアクセス制御 |
| `static-pages.spec.ts` | 静的ページ |

### スコープ外

- ドラッグ&ドロップによる並び替え (flaky になりやすいため)
- Admin ページ (テスト価値が低いため)
- プッシュ通知 (ブラウザ権限ダイアログが必要なため)
- Supabase Realtime のリアルタイム同期 (基本確認のみ、専用テストはスコープ外)

## アーキテクチャ

### ディレクトリ構成

既存構造を踏襲し、機能ごとに 1 ファイルの原則を維持する。

```
apps/web/e2e/
├── expenses.spec.ts       # 追加
├── polls.spec.ts          # 追加
├── reactions.spec.ts      # 追加
├── activity.spec.ts       # 追加
├── notifications.spec.ts  # 追加
├── souvenirs.spec.ts      # 追加
├── export.spec.ts         # 追加
├── print.spec.ts          # 追加
├── roles.spec.ts          # 追加
├── static-pages.spec.ts   # 追加
└── fixtures/
    └── auth.ts            # createExpenseViaAPI, createSouvenirViaAPI を追加
```

### fixtures 追加

高速セットアップのため、一部機能は API 直接呼び出しでデータ作成する。

```typescript
// セットアップ用 API ヘルパー (fixtures/auth.ts に追加)
createExpenseViaAPI(request, tripId, data)   // 経費テスト用
createSouvenirViaAPI(request, tripId, data)  // みやげテスト用
```

## テストケース詳細

### expenses.spec.ts (経費管理)

正常系:
- 経費を均等分担で追加できる
- 経費をカスタム分担で追加できる
- 経費を編集できる
- 経費を削除できる
- 精算サマリー (合計・明細) が表示される

異常系:
- カスタム分担の合計が経費金額と一致しない場合バリデーションエラーが表示される
- 必須項目 (タイトル・金額・支払者) 未入力でバリデーションエラーが表示される

### polls.spec.ts (日程投票)

正常系:
- メモを追加・編集できる
- 日程案を追加できる
- 日程案を削除できる
- ○/△/× で投票できる
- 投票を変更できる (○→△)
- 日程案を確定できる (旅行日程に反映される)

異常系:
- 確定済みの日程案は削除ボタンが非表示になる
- viewer ロールは日程案の追加・削除ボタンを表示しない

### reactions.spec.ts (候補へのリアクション)

正常系:
- 候補に 👍 リアクションを追加できる
- リアクションを 😕 に変更できる
- リアクションを削除できる

異常系:
- 共有ビュー (ログインなし) でもリアクション数が表示される

### activity.spec.ts (アクティビティログ)

正常系:
- アクティビティタブに切り替えてログが表示される
- スケジュール追加後にアクティビティログに記録される
- 「さらに読み込む」でページネーションできる

異常系:
- アクティビティがない場合に空状態メッセージが表示される

### notifications.spec.ts (通知・通知設定)

正常系:
- メンバー追加後に通知バッジが表示される (2ユーザー構成)
- 通知をクリックして既読にできる (バッジ数が減る)
- 「すべて既読」ですべての通知を既読にできる
- 設定ページで通知タイプごとにアプリ内通知をトグルできる

### souvenirs.spec.ts (みやげリスト)

正常系:
- みやげを基本情報 (品名のみ) で追加できる
- みやげに優先度・対象・メモを設定できる
- チェックボックスで購入済みにトグルできる (購入済みセクションに移動)
- みやげを編集できる
- みやげを削除できる
- 選択モードで一括削除できる
- 優先度順/作成順でソートできる

異常系:
- 品名未入力でバリデーションエラーが表示される

### export.spec.ts (エクスポートページ)

正常系:
- フォーマットを Excel から CSV に切り替えられる
- 「全選択」で全フィールドが選択される
- 「選択解除」で全フィールドの選択が解除される
- CSV 選択時に CSV 設定セクションが表示される
- ファイル名入力欄が旅行タイトルで自動入力される
- フィールド選択後にプレビューテーブルが表示される
- エクスポートボタンでダウンロードが起動する

異常系:
- フィールド未選択時に「出力する列を選択するとプレビューが表示されます」が表示される
- CSV 選択時にシート分けオプションが無効化される

### print.spec.ts (印刷ページ)

正常系:
- 印刷ページに旅行タイトル・日程が表示される
- スケジュールが日程別のセクションに表示される
- 「印刷 / PDF保存」ボタンが存在する

### roles.spec.ts (権限ベースアクセス制御)

viewer の制限:
- スケジュール追加ボタンが表示されない
- 候補追加ボタンが表示されない
- 経費追加ボタンが表示されない
- みやげ追加ボタンが表示されない
- 共有リンク生成ボタンにアクセスできない
- ポーリングに投票できる (viewer も可)
- 候補にリアクションできる (viewer も可)

editor の権限:
- スケジュールを追加・編集・削除できる
- 経費を追加・編集・削除できる

owner 専用:
- 旅行の基本情報を編集できる
- メンバーの追加・ロール変更・削除ができる

### static-pages.spec.ts (静的ページ)

正常系 (ナビゲーションなし、直接 URL アクセス):
- ランディングページ (`/`) が表示される
- FAQ ページ (`/faq`) が表示される
- 利用規約ページ (`/terms`) が表示される
- プライバシーポリシーページ (`/privacy`) が表示される
- ニュースページ (`/news`) が表示される

## 特殊事項

### テスト前提条件

- `expenses.spec.ts` / `souvenirs.spec.ts`: `createTripViaUI` が旅行日程を自動生成するため追加設定不要
- `polls.spec.ts`: 投票タブは旅行作成後すぐアクセス可能
- `notifications.spec.ts`: 2ユーザー構成 (`members.spec.ts` の browser context パターンを踏襲)
- `roles.spec.ts`: `members.spec.ts` の multi-user セットアップ (User A が User B を招待してロール設定)

### ファイルダウンロードのテスト

`export.spec.ts` では Playwright の `page.on('download', ...)` でダウンロードイベントを捕捉。
ファイル内容の検証はスコープ外 (ダウンロードが起動することのみ確認)。

### 印刷のテスト

`print.spec.ts` では `window.print()` のモックは不要。ページ内容の表示検証に集中する。

### テストパターンの踏襲

既存の確立されたパターンを維持する:
- `test.use({ storageState: ... })` による認証セッション再利用
- `getByRole()` / `getByLabel()` による accessibility-first セレクター
- トースト通知による操作成功確認
- 各テスト独立したユーザー/データ (共有ステートなし)

## 完成定義

- 10 ファイルすべてのテストが `bun run test:e2e` で green になること
- 既存の 62 テストが引き続き green であること
- 新規テストが flaky でないこと (CI 環境で安定して通過)
