# i18n 対応設計

## 概要

旅行計画アプリ sugara に多言語対応 (日本語 + 英語) を導入する。
英語圏の友人と旅行計画を共有するユースケースを想定。

## 全体の流れ

1. **Step 1**: next-intl を導入し、ホーム画面のハードコード文字列をメッセージファイルに切り出す
2. **Step 2**: 動作確認後、残り全画面の文字列を切り出す
3. **Step 3**: 英語翻訳ファイルを作成し、言語切り替え UI を追加

## スコープ

- **言語**: 日本語 (デフォルト) + 英語
- **対象**: UI の文字列 (ボタン、ラベル、トースト、エラーメッセージ等)
- **対象外**: ニュース記事 (markdown)、FAQ データ、ユーザー入力データ、API エラーメッセージ (`ERROR_MSG`)、プッシュ通知 (`PUSH_MSG`)

## 技術選定

- **next-intl**: Next.js App Router の i18n デファクト。RSC 対応が成熟
- **Cookie ベース**: URL は変えない (`/en/trips` のようなパスは使わない)
- Middleware なし: Cookie ベースで URL ルーティングしないため不要。`i18n/request.ts` の `getRequestConfig` 内で Cookie → `Accept-Language` の順にフォールバック
- 初回アクセス時は `Accept-Language` ヘッダーで自動判定
- 手動切り替え UI を提供、設定は Cookie (`NEXT_LOCALE`) に保存

## next-intl の設定

### next.config.ts

既存の `withSerwistInit` と `createNextIntlPlugin` を合成する:

```ts
import createNextIntlPlugin from "next-intl/plugin";
import withSerwistInit from "@serwist/next";

const withNextIntl = createNextIntlPlugin();

const nextConfig: NextConfig = { ... };

export default withNextIntl(
  withSerwistInit({ ... })(nextConfig)
);
```

### i18n/request.ts

ロケール解決ロジック:

```ts
import { getRequestConfig } from "next-intl/server";
import { cookies, headers } from "next/headers";

export default getRequestConfig(async () => {
  // 1. Cookie (手動切り替え済みの場合)
  const cookieLocale = (await cookies()).get("NEXT_LOCALE")?.value;
  // 2. Accept-Language ヘッダー (初回アクセス)
  const acceptLang = (await headers()).get("accept-language");
  const browserLocale = acceptLang?.startsWith("en") ? "en" : "ja";
  const locale = cookieLocale ?? browserLocale;

  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  };
});
```

### layout.tsx の `<html lang>` 動的化

`lang="ja"` のハードコードを next-intl の `getLocale()` で動的に設定:

```tsx
import { getLocale } from "next-intl/server";

export default async function RootLayout({ children }) {
  const locale = await getLocale();
  return <html lang={locale} suppressHydrationWarning>...</html>;
}
```

### 型安全性

`global.d.ts` で Messages 型を定義し、翻訳キーの typo をコンパイル時に検出:

```ts
import type messages from "./messages/ja.json";

declare module "next-intl" {
  interface AppConfig {
    Messages: typeof messages;
  }
}
```

## 既存 MSG オブジェクトとの関係

### 移行方針

- `MSG` の内容を next-intl のメッセージファイルに段階的に移行
- 移行済みのキーは `MSG` から削除し、import 元を `useTranslations` に切り替え
- 最終的に `MSG` オブジェクトは空になり、削除する

### 関数型メッセージの変換

`MSG` の関数型メッセージは ICU MessageFormat に変換:

```
// Before (TypeScript)
TRIP_BULK_DELETED: (n: number) => `${n}件の旅行を削除しました`

// After (ICU MessageFormat in ja.json)
"tripBulkDeleted": "{count}件の旅行を削除しました"
```

### ラベル定数群の移行

`CATEGORY_LABELS`, `STATUS_LABELS` 等のラベル定数もメッセージファイルに移行:

```json
{
  "labels": {
    "category": {
      "sightseeing": "観光",
      "restaurant": "飲食",
      "hotel": "宿泊"
    },
    "status": {
      "scheduling": "日程調整中",
      "draft": "下書き"
    }
  }
}
```

### 移行対象外

- `ERROR_MSG`: API レスポンス用。英語固定で i18n 不要
- `PUSH_MSG`: API 側で使用。将来的に対応検討するが今回は対象外
- `AUTH_ERROR_MAP` / `translateAuthError`: Step 2 で `labels` ネームスペースに移行

## Step 1: ホーム画面の文字列切り出し

### 対象ファイルと文字列

**`apps/web/app/page.tsx`** (20 strings):
- Hero: タイトル、説明文
- Auth ボタン: 新規登録、ログイン
- FEATURES 定数: 6 カード x (title + description) = 12 strings
- Footer: よくある質問、お知らせ、利用規約、プライバシーポリシー

**`apps/web/app/layout.tsx`**:
- `lang="ja"` → `getLocale()` で動的化
- metadata の description (OGP 含む)

**`packages/shared/src/messages.ts`**:
- `MSG.AUTH_SIGNUP_DISABLED` → メッセージファイルに移行

### 実装手順

1. `next-intl` をインストール (`bun add --filter @sugara/web next-intl`)
2. `apps/web/i18n/request.ts` を作成 (ロケール解決ロジック)
3. `apps/web/next.config.ts` に `createNextIntlPlugin` を追加 (`withSerwistInit` と合成)
4. `apps/web/messages/ja.json` を作成 (ホーム画面分)
5. `apps/web/global.d.ts` に Messages 型定義を追加
6. `layout.tsx` の `lang="ja"` を `getLocale()` で動的化、`NextIntlClientProvider` を追加
7. `page.tsx` のハードコード文字列を `getTranslations` に置き換え
8. `layout.tsx` の metadata を `getTranslations` で動的生成

### テスト

- next-intl のテスト用ユーティリティ: `NextIntlClientProvider` でラップするヘルパーを作成
- 既存テストへの影響: `NextIntlClientProvider` が必要になるため、テストユーティリティを更新
- メッセージキー整合性テスト: `ja.json` と `en.json` のキーが一致することを検証するテストを Step 3 で追加

## Step 2: 全画面の文字列切り出し

Step 1 の動作確認後、残りの画面を順次対応。

### 対象ルートグループ

- `(authenticated)/` — デスクトップ向けレイアウト
- `(sp)/sp/` — スマートフォン向けレイアウト (20 ページ)
- `auth/` — 認証画面
- `shared/` — 共有旅行ビュー (認証不要、`Accept-Language` で自動判定)
- `polls/shared/` — 共有投票ビュー
- `p/` — クイック投票の公開ビュー

### 対象画面リスト

- 認証画面 (login, signup, guest)
- ホーム (authenticated + sp)
- 旅行一覧/詳細/印刷/エクスポート
- スケジュール管理 (候補、パターン)
- 費用管理・精算
- 投票 (日程調整 + クイック投票)
- フレンド (一覧、申請、グループ)
- ブックマーク (リスト、詳細)
- 設定 (プロフィール、パスワード)
- お土産リスト
- ルーレット
- ナビゲーション (nav-links, bottom-nav, header-menu)
- 共有ビュー (旅行、投票)
- ダイアログ/モーダル類
- トーストメッセージ (MSG オブジェクトから移行)
- ラベル定数群 (CATEGORY_LABELS 等、23 ファイル 69 箇所)
- `AUTH_ERROR_MAP` / `translateAuthError`
- アクティビティログ (`ACTION_TEMPLATES`)
- 日付フォーマット (相対時間: "たった今", "分前" 等)

### MSG 利用箇所の規模

- `from "@/lib/messages"` の import: 85 ファイル
- ラベル定数の利用: 23 ファイル 69 箇所

## Step 3: 英語翻訳 + 言語切り替え UI

- `apps/web/messages/en.json` を作成
- 言語切り替えボタンを設定画面とヘッダーメニューに配置
- Cookie (`NEXT_LOCALE`) に保存
- `Accept-Language` による初回自動判定 (i18n/request.ts で実装済み)
- 共有旅行ビュー (`/shared/[token]`): Cookie 未設定の場合は `Accept-Language` で自動判定
- date-fns のロケール連動: `format` の `locale` オプションを現在のロケールに合わせる
- メッセージキー整合性テスト: `ja.json` と `en.json` の全キーが一致することを検証
